# Glassbox-Go Security & Structural Audit

## Observations & Findings

### 1. `limitedRuntime` Eviction Race Condition (`runtime/engine.go`)
- **Hypothesis**: The LRU cache eviction logic in `GetInstance` for memory-limited instances (`e.limitedRTs`) had a critical race condition. When a `limitedRuntime` is evicted, it's marked with `atomic.StoreInt32(&oldLRT.evicted, 1)`. If `oldLRT.activeReqs == 0`, it was safely closed.
- **The Catch**: In the previous version, `e.limitedCache[memLimit]` was accessed without verifying if the underlying `limitedRuntime` actually matched the one we just retrieved (`currentLRT`). This meant a cached compiled module could be returned for a completely different (and potentially closed) runtime if it had the exact same memory limit key, leading to "source module must be compiled before instantiation" panics.
- **Fix applied**: Ensuring `e.limitedRTs[memLimit] == currentLRT && e.limitedCache[memLimit] != nil` before retrieving the cached compiled module. This fixed `TestGetInstanceRace`.

### 2. Context Propagation Bug in Code Generator (`generator/generator.go`)
- **Hypothesis**: Sandboxed Wasm proxy functions did not correctly evaluate security gates (like `CheckFileAccess`) because they were not receiving the right context.
- **The Catch**: The code generator incorrectly hardcoded `context.Background()` when preparing `guestCallArgs` in the `guest` module generation logic, and more problematically, if we're not careful, the proxy might evaluate gates on a fresh context. Wait, the issue I was exploring with `TestPDFProcessorFilesystemGate` failing on "Default-Deny blocks file access when limits are not configured" suggests that the `CheckFileAccess` in the *guest* Wasm was being called with `context.Background()`.
- **Deeper Look**: Yes! The guest Wasm module generated by `generator.go` contains:
  `ret0, err = pdfprocessorImpl.ExtractTextFromFile(context.Background(), gapi.SandboxPath(args.Arg0))`
  If the user's *guest* implementation (`ExtractTextFromFile` inside Wasm) calls `gate.CheckFileAccess(ctx, string(path))`, it will ALWAYS fail, because `context.Background()` is passed, which has no `ActiveLimits` attached. ActiveLimits are a *host-side* concept!
- **Architectural contradiction**: The `README.md` and `pdf_processor.go` imply that the Guest implementation of the interface (which is pure Go code compiled to Wasm) should check `gate.CheckFileAccess`. But the Guest has NO limits configured! The limits are only on the host proxy! Wait, `pdf_processor.go` is the *user's real implementation*. It is what gets compiled to Wasm. In `pdf_processor.go`:
  ```go
  func (p *PDFProcessorImpl) ExtractTextFromFile(ctx context.Context, path gapi.SandboxPath) (string, error) {
      gate := &gapi.SecurityGate{}
      if err := gate.CheckFileAccess(ctx, string(path)); err != nil { ... }
  }
  ```
  This is executed INSIDE Wasm. But the limits are attached to the `ctx` on the HOST via `p.limits` in the generated proxy. The guest Wasm has NO limits in its context. This is a massive architectural flaw in the demo/examples or the framework's serialization strategy. `context.Context` is NOT serialized across the Wasm boundary! Therefore, the guest can NEVER evaluate `SecurityGate` correctly if it relies on context.
  Wait... `api/security_gate.go` checks limits. Does the Wasm guest have `SandboxLimits`? No.
  BUT... `VirtualHTTPClient` relies on `SecurityGate` and is ONLY used in the host. Wait, `fetch_http` host function uses `VirtualHTTPClient`.
  For filesystem access, WASI provides virtual filesystem capability. The host limits what WASI directories are mounted. The guest's `os.ReadFile` simply works if WASI allows it. The guest *should not* need to run `gate.CheckFileAccess`!
  Yet `examples/pdf/pdf_processor.go` explicitly runs `gate.CheckFileAccess`! Why?

## Most Important Issue to Focus On
The **Context/Capability Boundary Paradox** between Host and Guest.
Users (like the `examples/pdf/pdf_processor.go`) are told or shown to invoke `gapi.SecurityGate{}.CheckFileAccess(ctx, path)` inside the Wasm guest. However, `context.Context` does not cross the boundary, so `GetActiveLimits(ctx)` inside Wasm returns `nil`, triggering the Default-Deny rule.
We need to determine whether:
1. `SecurityGate` is purely a Host-side construct (meaning the guest should never use it, and the example is just wrong).
2. The code generator should inject Host-side validation *before* calling the guest, which it already does for `gapi.SandboxPath`!
