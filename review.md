The problem is that the example guest Wasm logic included code to evaluate `gapi.SecurityGate{}.CheckFileAccess(ctx, string(path))`. But this check fails by default because the limits are only configured in the *host's* context, and `context.Context` is NOT serialized across the boundary! The Guest Wasm doesn't have the `SandboxLimits`!
Furthermore, `gapi.SandboxPath` already forces a *host-side* validation of the path before crossing the boundary, making a guest-side check redundant!
Removing the manual `CheckFileAccess` from the guest logic fixes the problem and aligns perfectly with the architectural diagram where host capabilities manage security, not guest capability (which has none). I also reverted a bad `generator.go` change from earlier, maintaining proper syntax where guests receive `context.Background()`.
